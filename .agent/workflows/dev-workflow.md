---
description: 標準四段式開發工作流：計畫 → 執行 → 驗證 → 部署
---

## 說明
所有開發工作**必須**遵循以下四個階段。每個階段結束後，AI 必須停下來等待使用者審核，確認後才能進入下一階段。

---

## 第一階段：計畫 (PLAN)
1. 閱讀相關原始碼，理解現況
2. 建立或更新 `implementation_plan.md` 計畫文件，內容包含：
   - 根本原因分析
   - 計畫修改的檔案與方法
   - 不修改的部分，以及理由
   - 下一階段的驗證步驟
3. **停止 → 透過 `notify_user` 請使用者審核計畫文件**
4. 等待使用者明確確認（例如：「執行」、「沒問題」）後，才能進入第二階段

---

## 第二階段：執行 (EXECUTE)
1. 按照已核准的計畫精確修改程式碼
2. **不得偏離計畫** — 若發現預料外的複雜度，則返回第一階段重新計畫
3. 所有修改完成後，更新計畫文件標記已完成項目
4. **停止 → 列出所有修改的檔案與 diff，請使用者審核代碼**
5. 等待使用者確認後，才能進入第三階段

---

## 第三階段：驗證 (VERIFY)
1. 啟動本地開發伺服器：`npx vite-node server.ts`
2. 逐一測試每個受影響的 API 端點：
   - 用 `read_url_content` 呼叫 `http://127.0.0.1:3000/api/...`
   - 確認回傳格式符合預期 Schema
   - 確認沒有破壞現有功能（regression）
3. 使用瀏覽器工具開啟 `http://127.0.0.1:3000`，目視確認受影響的 UI 頁面
4. 將測試結果記錄於 `walkthrough.md`
5. 測試完成後關閉本地伺服器
6. **停止 → 透過 `notify_user` 呈現驗證結果給使用者**
7. 等待使用者確認結果可接受後，才能進入第四階段

---

## 第四階段：部署 (DEPLOY)
只有在使用者明確確認第三階段結果後才能執行。

// turbo
1. `git add .`
// turbo
2. `git commit -m "<類型>: <本次修改的簡短說明>"`
   - 類型：`fix:`（修正）、`feat:`（新功能）、`perf:`（效能）、`refactor:`（重構）、`chore:`（雜務）
3. **停止 → 顯示 commit 訊息與修改清單，請使用者確認是否推送**
4. 等待使用者明確同意後，才執行：
// turbo
5. `git push`
6. 確認推送成功，並備註 GitHub commit 連結

---

## 規則
- **絕不跳過任何階段**
- **絕不在使用者明確同意前執行 `git push`**（第四階段步驟 3 審核是強制的）
- 若使用者在未完成前置階段的情況下要求「推送」，必須提醒工作流順序
- 即時端點（`live/top100`、`live/border`）的快取設定必須使用 `s-maxage ≤ 60`
- 歷史資料端點（`event/[id]/top100`、`event/[id]/border`）可使用 `s-maxage=86400`
